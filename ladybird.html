§ 01

Abstract

Browser fingerprinting identifies users by aggregating device-specific signals across dozens of JavaScript APIs — canvas pixel output, WebGL renderer strings, audio oscillator waveforms, high-resolution timestamps, navigator properties, worker behaviour, and more. Most anti-fingerprinting implementations patch these surfaces at the JavaScript layer: random noise injected per call, or static spoofed strings returned verbatim. Both approaches fail under sustained ML-based detection. Per-call randomness creates temporal inconsistency (real hardware produces identical fingerprints across sessions). Static strings produce cross-surface incoherence (a claimed Apple M2 GPU alongside a Windows ANGLE renderer is immediately flagged).

This project builds a complete anti-fingerprinting engine as a C++ layer inside a forked Ladybird browser. Every API surface is intercepted at the component level — not at the JavaScript boundary — and redirected through a unified identity system: HexEdits::FingerprintEngine. The engine generates a single, internally consistent hardware profile at startup and derives all API responses from that profile deterministically, producing the same fingerprint across every session, every call, and every surface simultaneously.

No surface is a JavaScript patch layer — every interception occurs at the C++ component that implements the API.

§ 02

Architecture

The engine is structured in three layers: a Device Registry that holds real-world device templates, a Profile Generator that assembles them into a coherent BrowserProfile, and a set of Spoof singletons in the LIBWEB layer that intercept API calls and return profile-derived values. 

/* Startup sequence — called once from DeviceProfile::initialize() */

DeviceProfile::initialize()
  → DeviceRegistry::select_random_device()     // pick a real device template
  → ProfileGenerator::generate_profile()          // build BrowserProfile from template
  → ProfileGenerator::initialize_spoof_systems()   // arm all spoof singletons

/* At API call time (any surface) */

// e.g. navigator.hardwareConcurrency
WorkerSpoof::get_hardware_concurrency()
  → ProfileGenerator::the().current_profile()
  → profile.worker_profile().hardware_concurrency

Central Data Structure: BrowserProfile

All profile state lives in a single BrowserProfile instance owned by ProfileGenerator. Every spoof singleton holds a const reference to a sub-spec extracted from this profile. The profile is generated once and never mutated at call time — all randomness is pre-computed at initialization.

Randomness Model

The engine uses two distinct sources of randomness with different lifetimes. At profile generation time, a u64 device_seed is derived from the selected device template's identifier using a polynomial hash over its byte sequence. All pre-computed patterns (audio noise, FFT offsets, timing jitter sequences) are generated from this seed using a Linear Congruential Generator — same seed produces identical patterns forever, simulating a real device that always returns the same fingerprint. At call time, a RandomGenerator singleton provides session-unique jitter using a separate seed generated at startup, ensuring temporal variance within a session without cross-session inconsistency.

Design Principle

Real hardware is deterministic: an iPhone 15 Pro Max running Safari always produces the same AudioContext fingerprint, the same canvas pixel output, the same WebGL renderer string. The engine models this by separating identity (derived from device seed, stable forever) from timing noise (session-unique, varies per call).

§ 03

Platform Coherence Enforcement

The most detectable anti-fingerprinting failure is cross-surface incoherence: a browser claiming a macOS Apple M2 GPU in WebGL while its canvas rendering config describes Windows subpixel rendering, or a navigator profile listing 32 hardware threads for a device that WebGL identifies as a mobile Adreno GPU. The engine enforces coherence through a validation step in ProfileGenerator that runs after profile assembly.

OS Inference from WebGL Renderer

The validation begins by inferring the claimed OS from the WebGL UNMASKED_RENDERER_WEBGL string using a priority-ranked ruleset. ANGLE/Direct3D renderer strings imply Windows. Mesa DRI or llvmpipe implies Linux. Apple M or Metal implies macOS. Adreno, Mali, or PowerVR implies Android. Apple A-series GPU strings (A14 GPU, A17 GPU) imply iOS. The inferred OS is then cross-checked against the canvas profile's expected_os field. 

String ProfileGenerator::infer_os_from_webgl_renderer(
    String const& renderer, String const& vendor)
{
    // Priority 1: renderer string is authoritative
    if (renderer.contains("Apple M"sv) && renderer.contains("GPU"sv))
        return "macOS"_string;              // Apple M1/M2/M3/M4

    if (renderer.contains(" A1"sv) && renderer.contains("GPU"sv))
        return "iOS"_string;               // Apple A15/A16/A17

    if (renderer.contains("Mesa DRI"sv) || renderer.contains("llvmpipe"sv))
        return "Linux"_string;

    if (renderer.contains("ANGLE"sv) && renderer.contains("Direct3D"sv))
        return "Windows"_string;

    if (renderer.contains("Adreno"sv) || renderer.contains("Mali"sv)
        || renderer.contains("PowerVR"sv))
        return "Android"_string;

    return "Unknown"_string;
}

Each constraint is checked and, on failure, the profile is corrected to match the inferred OS rather than rejecting the profile outright — this makes the system robust to partially-specified device templates.

§ 04

Canvas & WebGL

Canvas and WebGL fingerprinting both rely on extracting pixel-level output from the GPU rendering pipeline. The same image rendered on different hardware produces subtly different pixel values due to GPU-specific shader compilation, rounding in floating-point blending operations, and font rasterization differences. The engine intercepts both APIs at the C++ layer before data reaches JavaScript.

Canvas Pixel Modification

The CanvasSpoof interceptor wraps getImageData() and toDataURL(). After the real pixel data is computed, a deterministic per-pixel modification is applied using the device seed and the pixel's (x, y) position as inputs to the LCG. The modification magnitude is small (±1–3 per channel) to avoid visual artifacts while producing a unique pixel signature. The pattern is pre-computed into a lookup table at initialization — no per-call computation beyond table lookup.

WebGL Parameter Spoofing

WebGLSpoof overrides all fingerprint-relevant getParameter() calls. The WEBGL_debug_renderer_info extension returns the profile's unmasked_renderer and unmasked_vendor strings. Precision parameters (HIGH_FLOAT, MEDIUM_FLOAT, HIGH_INT) are set to values matching the claimed GPU class. Max texture size, max viewport dimensions, and max vertex attributes are taken from real device measurements stored in the device template.

Known Limitation

Shader compilation timing via KHR_parallel_shader_compile is not currently intercepted. A fingerprinter measuring compile time for a known shader can infer the true GPU class even if the renderer string is spoofed. This is the primary remaining canvas/WebGL detection vector.

§ 05

Audio Fingerprint

The Web Audio API fingerprint is constructed by routing a triangle wave oscillator through a DynamicsCompressor and sampling the output into an AnalyserNode. The resulting FFT frequency bins produce a signature that varies by sample rate, compressor parameters, and CPU/GPU floating-point rounding — all device-specific. The engine's first approach used per-call random noise, which reduced detection rates from 75% to 45% according to testing. The architecture comment documents the remaining problem:

From AudioFingerprint.h

Random noise each time → ML detected temporal inconsistency. This approach: deterministic signature based on device profile → same fingerprint always. Detection reduction: 75% → 45%.

The current implementation generates two pre-computed arrays from the device seed at initialization time and caches them for the lifetime of the browser session.

Oscillator Noise Pattern

A 48,000-sample noise pattern (one second at 48kHz) is generated using a Linear Congruential Generator seeded from the device identifier. Each sample is a floating-point value in the range [-0.000015, +0.000015] — matching the small noise floor of real audio hardware. The pattern repeats modulo 48,000 for any sample index beyond the first second, simulating real hardware's periodic noise floor characteristic. 

/* LCG parameters from Numerical Recipes — deterministic across all runs */
u32 rng_state = static_cast<u32>(seed & 0xFFFFFFFF);
for (size_t i = 0; i < 48000; ++i) {
    rng_state = rng_state * 1103515245 + 12345;
    float normalized = static_cast<float>(rng_state & 0x7FFFFFFF)
                      / static_cast<float>(0x7FFFFFFF);
    // Scale to ±0.000015 — matches real hardware noise floor
    float noise = (normalized - 0.5f) * 0.00003f;
    m_oscillator_noise_pattern.append(noise);
}

FFT Bin Offsets

Real AnalyserNode implementations apply FFT processing with integer bin quantization. Different hardware produces slightly different bin values due to floating-point rounding in the FFT butterfly operations. The engine pre-computes 256 signed byte offsets (one per FFT bin, range [-2, +2]) from the upper half of the device seed. When getByteFrequencyData() or getFloatFrequencyData() is called, the C++ interceptor applies the per-bin offset to the real computed value before returning it, clamped to valid uint8 range.

The AudioSpec sub-profile also specifies the exact compressor parameters returned by the DynamicsCompressorNode property accessors: threshold, knee, ratio, attack, release, and reduction. These match values observed on real devices of the profiled type.

§ 06

High-Resolution Timing

performance.now(), performance.mark(), performance.measure(), and resource timing APIs are primary fingerprinting vectors: different browsers apply different precision coarsening, different OS schedulers introduce different jitter distributions, and CPUs accumulate measurable clock drift over a session. The engine intercepts these APIs in HighResolutionTimeSpoof and applies a four-layer transformation to the raw timestamp.

Layer 1 — Browser-Specific Coarsening

Three coarsening modes match the three major browser implementations. Chrome/Edge use Floor: the timestamp is simply floored to the nearest precision multiple. Firefox uses Round: nearest-multiple rounding. Safari uses Stochastic: a probabilistic rounding where the fractional part of the timestamp (relative to the precision interval) determines the probability of rounding up, making the output non-deterministic per call but statistically correct in aggregate. 

double HighResolutionTimeSpoof::stochastic_round(double value, double precision)
{
    double floored = floor(value / precision) * precision;
    double fraction = (value - floored) / precision;   // fraction in [0, 1)

    // Deterministic "random" decision — same call number = same rounding
    u64 decision_seed = m_jitter_seed + m_coarsen_call_count;
    u32 rng = static_cast<u32>((decision_seed * 1103515245 + 12345) & 0xFFFFFFFF);
    double random = static_cast<double>(rng & 0x7FFFFFFF)
                  / static_cast<double>(0x7FFFFFFF);

    // P(round up) = fraction — Safari's actual algorithm
    return (random < fraction) ? (floored + precision) : floored;
}

Layer 2 — Deterministic Jitter

A 10,000-sample jitter sequence is pre-generated at initialization using Box-Muller transform applied to LCG output, producing a Gaussian-distributed sequence with the profile's jitter_amplitude_ms as the 3-sigma bound. Each call to a timing API consumes the next sample from this sequence, wrapping at 10,000. The scheduler jitter (OS-level variance) uses samples from the upper half of the sequence (offset +5000), keeping the two noise sources uncorrelated.

Layer 3 — CPU Clock Drift

Real CPUs drift from their nominal frequency due to temperature variation, voltage scaling, and aging. The profile stores a cpu_clock_drift_ppm value specific to the device class (typical range: 10–30 ppm). Drift is computed as elapsed session time in seconds multiplied by the ppm rate, converting to a millisecond offset that accumulates across the session. A long-running session on the same device produces a measurably different timestamp sequence than a fresh session — which is exactly what real hardware does.

Layer 4 — API-Specific Noise

performance.mark() timestamps receive an additional mark_precision_variance noise term. performance.measure() durations receive a separate measure_duration_noise term applied to the raw start-to-end difference. Resource timing APIs receive a larger network_timing_jitter_ms term (network timings are inherently noisier than CPU timings and ML models are trained on this difference).

§ 07

Navigator & Worker Spoofing

The navigator API exposes more fingerprinting surface than any other single interface: hardware concurrency, device memory, user agent, platform string, language list, online status, cookie enabled, PDF viewer, plugin list, touch points, battery status, network information, and media device counts. Every field is derived from the NavigatorProfile and must be internally consistent. The engine selects from a database of real device profiles — not synthetic combinations — each modeled on a specific device and browser version.

Real Device Profile Examples

Sample entries from the navigator profile database illustrate the level of specificity. A "Windows 11 x64 - Chrome - Intel i7" profile sets hardwareConcurrency to 16, deviceMemory to 8, platform to "Win32", maxTouchPoints to 0, pdfViewerEnabled to true, battery API charging state to true with infinite discharging time (desktop, always plugged in), and NetworkInformation API disabled (Chrome on Windows desktop does not expose it). A "Android 14 - Galaxy S24 Ultra" profile sets hardwareConcurrency to 8, deviceMemory to 12, battery discharging true, connection API enabled with effectiveType: "4g", and maxTouchPoints to 10.

Worker Consistency Enforcement

Dedicated Workers, Shared Workers, and Service Workers expose a subset of navigator — hardwareConcurrency, userAgent, languages, and deviceMemory. A common detection vector is reading these properties from both the main thread and a worker thread and checking for mismatches. The engine routes all worker API calls through WorkerSpoof, which reads directly from the same NavigatorProfile that the main thread uses, ensuring byte-for-byte consistency: 

/* WorkerSpoof — all values sourced from the same profile as the main thread */
u32 get_hardware_concurrency()
  → profile.worker_profile().hardware_concurrency      // matches main thread

String get_worker_user_agent()
  → profile.navigator_profile().user_agent             // identical string

u64 get_device_memory_gb()
  → profile.navigator_profile().device_memory_gb       // same value

Worker spawn timing is also spoofed. apply_worker_spawn_delay() calls usleep() for a duration sampled from the profile's dedicated_worker_spawn_time_min/max range. iOS Safari profiles set very short spawn times (180ms base) with low variance. Android high-end profiles use 220ms with higher variance due to big.LITTLE architecture scheduling. iOS profiles hard-set supports_shared_workers = false (a real Safari limitation) and supports_wasm_threads = false.

Animation Frame Timing

requestAnimationFrame timestamp variance is controlled by AnimationTimingSpec. Frame jitter is computed with a continuous exponential decay function — not discrete performance buckets — from the CPU's single-core benchmark score: jitter = 2.5 × exp(−0.00095 × cpu_score), clamped to [0.1ms, 2.5ms]. This ensures Benford's Law compliance in the timestamp digit distribution, which ML classifiers specifically test for. Dropped frame probability uses a logistic function of CPU score, with a GPU vendor multiplier (mobile Adreno/Mali adds 50% more drops than desktop NVIDIA) and a small hash-derived perturbation per vendor string to avoid identical curves across profiles.

§ 08

Remaining Surfaces

Clipboard

ClipboardProfileDatabase maps each platform to its native clipboard format vocabulary. macOS profiles return NSPasteboardTypeString, NSHTMLPboardType, and NSPasteboardTypePNG as native format strings. Windows profiles return CF_UNICODETEXT, CF_HTML, and "PNG". iOS profiles use UTI strings: public.utf8-plain-text, public.html, public.png. Operation timing is also platform-specific: macOS read operations average 1.5ms, Windows average 2.5ms (more IPC overhead), Android averages 3.0ms. iOS disables supports_web_custom_formats and caps max_clipboard_size_bytes at 50MB.

Compression Streams

CompressionStream and DecompressionStream timing is fingerprinted through the per-chunk latency, flush latency, and output-to-input ratio. CompressionSpoof intercepts each compress_and_enqueue_chunk() and decompress_and_enqueue_chunk() call with a simulated delay: base_latency_us + (compress_latency_per_kb × input_KB) + jitter. A thermal drift model accumulates sustained compression load: after burst_threshold_ops consecutive operations, each subsequent call's latency is multiplied by 1 + (sustained_time_sec × thermal_drift_rate), simulating CPU throttling under load.

Geometry Precision

CSS transforms, SVG path operations, and DOMMatrix arithmetic use floating-point matrix math whose precision varies by GPU and platform. GeometrySpoof wraps the three critical operations: matrix multiply (apply_multiply_error), matrix inverse (apply_inverse_error), and point transform (apply_transform_error). Each applies a small random error in the range [-error_rate, +error_rate] drawn from the session-unique RNG, while apply_precision() rounds all geometry values to the profile's fp_epsilon (e.g. 1e-14 for Intel integrated, 1e-16 for NVIDIA or Apple Silicon).

Form Input & Behavioural Profiles

The FormInputProfile models human interaction at a level of detail intended to defeat bot detection systems that analyze input event patterns rather than static fingerprints. The KeystrokeTimingProfile specifies average inter-keystroke interval, variance coefficient, minimum/maximum hold duration, per-digraph timing overrides (e.g. "th" → 85ms, "qu" → 120ms), burst length distributions, and backspace frequency.

Data Isolation

ProfileDataIsolation implements per-profile directory separation for all browser storage: cookies, localStorage, sessionStorage, IndexedDB, HTTP cache, service worker registrations, and permission grants. Each profile gets its own directory under base_profiles_path/profile_id/. A path cache keyed by storage type avoids repeated string concatenation. This ensures that switching between profiles produces zero state leakage — the browser behaves as a completely fresh instance with a different hardware identity.

GPC Spoofing
Global Privacy Control signal derived from GPCSpec.enabled. Platform-specific: enabled on privacy-focused profiles, disabled on others to match population base rates.
Geolocation
Accuracy variance and request delay timing are profile-specific. GPS accuracy varies by device class: mobile high-end gets ±5m, desktop gets ±50m (no GPS hardware).
Element Timing
Image, canvas, video, text, and form render timing baselines derived from GPU/CPU tier. Image render base: 10ms (high-end GPU) to 25ms (integrated). Canvas is consistently slower than image.
Cookie Behaviour
Per-platform cookie acceptance, SameSite behaviour, third-party restrictions, and storage quota are derived from the browser type and OS profile.

Design Limitations

The engine does not currently implement profile persistence across restarts — DeviceProfile::initialize() contains a TODO: Add persistence (load from disk) comment, and select_random_device() picks a fresh device each launch. This means the fingerprint is stable within a session but changes across browser restarts, which real hardware does not do. The correct fix — hashing the device selection to a stable seed stored on disk — is the primary outstanding work item.

Canvas spoofing at the pixel level is effective against standard fingerprinting scripts but does not address entropy in text rendering metrics (measureText() width variations) at sub-pixel precision. The text measurement cache covers common strings but cannot pre-cache arbitrary text. Fingerprinters that render novel text strings will get consistent measurements across repeated calls for the same string, but the absolute values will not match a real device.

Summary

The engine covers 15+ fingerprinting surfaces at the C++ component level with full cross-surface coherence enforcement. Every API response is derived from a single consistent hardware identity, generated once, and replicated deterministically across all calls and all surfaces simultaneously. The architecture's core insight — that consistency, not randomness, is the correct anti-detection primitive — distinguishes it from all JavaScript-layer approaches.
