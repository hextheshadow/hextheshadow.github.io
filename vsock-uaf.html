§ 01

Abstract

We identified and confirmed a use-after-free vulnerability in the Linux kernel's vsock loopback transport, present in kernel version 6.12.x and submitted as a research target for kernelCTF. The bug resides in virtio_transport_consume_skb_sent() within net/vmw_vsock/virtio_transport_common.c and is triggered by a race condition between the loopback workqueue and the socket teardown path.

The vulnerability yields a silent UAF write primitive — a controlled subtraction on a size_t field of a freed or live kernel object — and a UAF read that exposes raw kernel text and data pointers from freed socket memory. Neither primitive triggers KASAN on the tested configuration.

This research was conducted using a novel AI-augmented kernel research pipeline, in which an AI language model served as a continuous technical collaborator — reading source code, reasoning about kernel internals, proposing hypotheses, and generating GDB scripts and PoC code across an extended, iterative session. We document both the technical findings and the methodology.

§ 02

Discovery & Initial Hypothesis

Why vsock?

The vmw_vsock subsystem implements virtual socket communication between a host and its guests, and crucially — via the loopback transport — between processes on the same machine using VMADDR_CID_LOCAL. This loopback path is less audited than the mainline network stack, uses a dedicated workqueue (vsock-loopback), and manages socket-owned SKBs that carry a back-pointer (skb->sk) to the originating socket.

Source Navigation

The starting point was reading vsock_loopback.c and tracing the lifecycle of an SKB from allocation to delivery. The file structure of interest was:

FileRole
net/vmw_vsock/af_vsock.cSocket lifecycle, release/destruct path
net/vmw_vsock/vsock_loopback.cWorkqueue, pkt_queue, loopback send
net/vmw_vsock/virtio_transport_common.cconsume_skb_sent, close, credits

In vsock_loopback_work(), every SKB dequeued from pkt_queue passes through virtio_transport_consume_skb_sent(skb, false) before being delivered to the receiver. This function dereferences skb->sk without holding any reference to the socket: 

/* net/vmw_vsock/virtio_transport_common.c */
void virtio_transport_consume_skb_sent(struct sk_buff *skb, bool consume)
{
    struct sock *s = skb->sk;          /* ← raw pointer, no refcount held */

    if (s && skb->len) {
        struct vsock_sock  *vs  = vsock_sk(s);
        struct virtio_vsock_sock *vvs;

        vvs = vs->trans;                  /* ← may be freed */

        spin_lock_bh(&vvs->tx_lock);      /* ← UAF spin_lock  */
        vvs->bytes_unsent -= skb->len;    /* ← UAF WRITE      */
        spin_unlock_bh(&vvs->tx_lock);
    }
    ...
}

The hypothesis formed immediately: if the socket is closed (and vvs freed) while SKBs referencing it still sit in the loopback pkt_queue, the workqueue will dereference freed memory when it eventually runs.

§ 03

Vulnerability Deep Dive

struct virtio_vsock_sock — Memory Layout

struct virtio_vsock_sock is allocated via kzalloc(sizeof(*vvs), GFP_KERNEL) in virtio_transport_do_socket_init(). On x86-64 with kernel 6.12, its size is approximately 80 bytes, placing it in the kmalloc-96 SLUB cache.

OffsetFieldTypeNotes
+0x00rx_lockspinlock_t4 bytes
+0x08rx_queuesk_buff_headskb receive queue
+0x18tx_lockspinlock_tprotects tx fields
+0x20bytes_unsentsize_tUAF write target
+0x28tx_cntu32tx packet counter
+0x30peer_buf_allocu32peer's buffer size
+0x34peer_fwd_cntu32peer forward count
+0x38buf_allocu32local buffer size
+0x3cfwd_cntu32forward count
+0x40vsk*vsock_sockback-pointer

The write primitive lands at vvs+0x20 (bytes_unsent), subtracting skb->len (which we control via SEND_SIZE, typically 8192 bytes). This is a controlled subtraction on a freed size_t.

The Close Path — What kfree(vvs) Actually Requires

Reading the full teardown chain from source was essential. The path is non-trivial:

1
close(fd) → vsock_release() → __vsock_release()
Calls vsk->transport->release(vsk) = virtio_transport_release(). Sets SOCK_DEAD flag.
2
virtio_transport_close() — the branching point
Fast path: if peer_shutdown == SHUTDOWN_MASK → sends RST, returns true → immediate teardown.
Slow path: schedules close_work with VSOCK_CLOSE_TIMEOUT = 8 * HZ → returns false. kfree(vvs) deferred by 8 seconds.
3
sock_put(sk) → refcount drop → vsock_sk_destruct()
Calls vsock_deassign_transport() → vsk->transport->destruct(vsk) = virtio_transport_destruct().
4
kfree(vvs) — slot returns to kmalloc-96
Only now is vvs actually freed. vsk->trans = NULL set immediately after.

Critically: cancel_pkt is never called in this path. vsock_loopback_cancel_pkt() (which calls virtio_transport_purge_skbs() to drain the pkt_queue) is only invoked from vsock_connect_timeout and vsock_shutdown — not from __vsock_release. SKBs in the loopback queue survive the close path unaffected.

The Race Window

On a single-CPU QEMU instance (the kernelCTF environment), scheduling is cooperative between kernel threads. The loopback workqueue (vsock-loopback, created with flags=0) runs as a bounded workqueue worker. After close(fd), the kernel returns to userspace briefly — during which sched_yield() or natural scheduling causes the workqueue worker to run before the delayed close work fires. 

/* Timeline confirmed via GDB */
[user] send(fd, buf, 8192, MSG_DONTWAIT)  /* × 6 — SKBs queued in pkt_queue */
[user] close(fd)                          /* virtio_transport_close() → schedules 8s work */
                                          /*   vvs still ALIVE at this point              */
[wq]   vsock_loopback_work() wakes up
[wq]     consume_skb_sent(skb, false)
[wq]       s   = skb->sk           /* UAF READ  — sk still valid, refcount=0 */
[wq]       vvs = vs->trans         /* vvs pointer — still live               */
[wq]       vvs->bytes_unsent -= 8192  /* WRITE on live-but-closing vvs          */
[wq]     deliver pkt to receiver
[8s]   virtio_transport_close_timeout()
[8s]     virtio_transport_do_close()
[8s]     kfree(vvs)                /* ← freed HERE, not at close() time      */
Key Finding — Write Happens on Live Object
On the tested single-CPU configuration, the UAF write to bytes_unsent consistently occurs on a live but refcount=0 vvs, not a freed one. kfree(vvs) happens 8 seconds later (slow path) or immediately (fast path if peer already shut down). Forcing the fast path and reversing the ordering to achieve a true cross-cache write is the open exploitation challenge — detailed in §5.
§ 04

Confirmation via GDB

Hardware Watchpoint Strategy

We confirmed the bug using a live GDB session attached to the QEMU kernel via -s -S flags. The strategy: break on virtio_transport_consume_skb_sent, detect the UAF condition via sk->__sk_common.skc_refcnt.refs.counter == 0, then arm a hardware watchpoint on the identified vvs+0x20. 

# GDB script — UAF detection and watchpoint
break virtio_transport_consume_skb_sent
commands
  silent
  set $sk  = skb->sk
  set $ref = ((struct sock*)$sk)->__sk_common.skc_refcnt.refs.counter
  set $vvs = ((struct vsock_sock*)$sk)->trans
  set $len = skb->len
  if $len > 0 && $ref == 0
    set $last_uaf_vvs = $vvs
    printf "*** UAF HIT vvs=%p len=%d ***\n", $vvs, $len
    printf "offset32 = 0x%lx\n", *(size_t*)((char*)$vvs + 0x20)
  end
  continue
end

break virtio_transport_destruct
commands
  silent
  set $vvs_d = ((struct vsock_sock*)$rdi)->trans
  if $target_vvs == 0 && $vvs_d == $last_uaf_vvs
    set $target_vvs = $vvs_d
    printf "FREED vvs=%p — watchpoint armed\n", $target_vvs
    watch *(size_t*)((char*)$target_vvs + 0x20)
  end
  continue
end

Confirmed Findings
(1) The UAF write fires reliably — bytes_unsent decremented by exactly skb->len = 8192, multiple times per connection (once per SKB in flight).

(2) KASAN does not fire. The object is technically alive (refcount=0 but not yet freed) at write time — KASAN only instruments kfree()-ed memory, not refcount=0 objects.

(3) After kfree(vvs) (via virtio_transport_destruct), the freed slot's first 8 bytes contain a SLUB freelist pointer — confirming it entered the kmalloc-96 free list and is available for reclaim.

UAF Read — Kernel Pointer Leak

At the point consume_skb_sent dereferences skb->sk, the sock structure is being torn down with refcount=0. GDB inspection of adjacent fields revealed live kernel text and data pointers: 

/* sk at time of UAF read */
sk+0x28: 0xffffffff88dc23e0   /* sk_prot  — kernel text pointer (.text) */
sk+0x30: 0xffffffff8a40b700   /* sk_net   — kernel data pointer (.data) */

If these values can be exfiltrated to userspace, they provide a direct KASLR bypass without needing any cross-cache technique. The sk_prot pointer is a fixed offset from the kernel base; subtracting the known compile-time offset yields _text.

Spray Landing — add_key Confirmation

To confirm that the freed vvs slot is reclaimable, we sprayed 512 add_key("user", ...) calls with 72-byte payloads (→ kmalloc-96) immediately after kfree(vvs). GDB watchpoint confirmed the marker value landing: 

Hardware watchpoint: *(size_t*)((char*)$target_vvs + 0x20)
Old value = 0
New value = 3501   ← 0xdad — spray marker confirmed in freed slot ✓

Old value = 3501
New value = 0    ← kvfree_sensitive() zeroes temp payload buffer ✗

The marker confirmed cache alignment: the add_key temporary payload buffer lands in kmalloc-96, matching the freed vvs. However, kvfree_sensitive() at the end of the add_key syscall zeroes and frees the temporary buffer — it is not the persistent user_key_payload object. This drove the requirement to switch spray objects.

§ 05

Exploitation Primitives & Constraints

Spray Object Analysis

To convert the UAF write into a useful exploit primitive, a spray object must satisfy:

RequirementReason
Allocated in kmalloc-96Must land in same slab cache as freed vvs (size ≈ 80 → kmalloc-96)
Not zeroed on free/allocValue at +0x20 must persist after UAF write
Offset +0x20 readable/controlledUAF write target is fixed at bytes_unsent = vvs+0x20
No spinlock at +0x00consume_skb_sent calls spin_lock_bh(&vvs->tx_lock) — tx_lock is at +0x18; +0x00 is rx_lock
Stays alive after allocationspray → workqueue runs → UAF write → read back = three separate points in time

Candidate Objects

add_key temporary payload buffer — eliminated. kvfree_sensitive() zeroes it at end of syscall. Confirmed by GDB watchpoint.

user_key_payload — viable in principle. Allocated as kmalloc(sizeof(struct user_key_payload) + datalen) = 18 + datalen bytes. For kmalloc-96: datalen = 78. Persists in keyring until explicitly revoked. Not zeroed on free. Readable via keyctl(KEYCTL_READ). Problem: the temporary payload copy (also kmalloc(78) → kmalloc-96) competes for the same freed slot and is zeroed first.

msg_msg — the canonical technique (Popov et al.). Size: 0x30 header + 48 bytes data = 96 → kmalloc-96. Persistent in message queue. msg_msg.next at +0x20 — the exact write target. Normally NULL for short messages; UAF decrement corrupts it to 0 - 8192 = 0xffffffffffffe000. Following this in msgrcv() → OOB kernel read → KASLR bypass. Problem with this kernel: init_on_alloc=0 — fresh msg_msg allocations have stale data at +0x00 (m_list fields), making the spinlock at spin_lock_bh(&vvs->tx_lock) potentially hang or crash.

/* msg_msg layout — kmalloc-96 */
struct msg_msg {
    struct list_head m_list;     /* +0x00: 16 bytes — contains fwd/bck ptrs */
    long             m_type;     /* +0x10 */
    size_t           m_ts;       /* +0x18 */
    struct msg_msgseg *next;     /* +0x20 ← UAF WRITE LANDS HERE */
    void             *security;  /* +0x28 */
};                               /* data follows at +0x30 */

The Ordering Problem — Root Cause

The fundamental constraint is execution ordering on a single-CPU system. Our confirmed observation is that consume_skb_sent always fires on the live vvs (before kfree). For a cross-cache UAF write, we need: 

/* DESIRED ordering — cross-cache UAF */
[1] spray: msg_msg / user_key_payload allocated → blocked in msgsnd
[2] close(fd) → kfree(vvs) IMMEDIATELY (fast path)
[3] spray object reclaims freed vvs slot
[4] vsock_loopback_work runs → consume_skb_sent
    → vvs->bytes_unsent -= 8192
    → ACTUALLY hits msg_msg.next -= 8192   ← cross-cache write ✓

/* ACTUAL ordering observed */
[1] close(fd) → virtio_transport_close() → schedules 8s work (slow path)
[2] vsock_loopback_work runs → consume_skb_sent (write on LIVE vvs)
[3] [8 seconds later] kfree(vvs)
[4] spray → reclaims freed slot (too late, write already done)

Fast path analysis: virtio_transport_close() takes the fast path (immediate teardown, no 8s delay) only when vsk->peer_shutdown == SHUTDOWN_MASK at the time of close(fd). The peer's shutdown packet arrives via the same loopback workqueue as the data SKBs. The workqueue processes them in FIFO order — data SKBs first (sent before listener closed), shutdown packet second. By the time peer_shutdown == SHUTDOWN_MASK is set on the sender, consume_skb_sent has already run for all data SKBs.

This is the core catch-22: you cannot get peer_shutdown set without the workqueue running, and the workqueue running means consume_skb_sent has already fired on the live vvs. Reversing this ordering requires either a multi-CPU environment (workqueue deferred to a different CPU), a way to hold the workqueue (flush/cancel from userspace has no syscall interface), or a different trigger mechanism entirely.

Current Primitive Summary
Confirmed: Reliable controlled subtraction (-= skb->len) on vvs->bytes_unsent at vvs+0x20, firing once per SKB in flight (typically 6 writes per connection attempt), KASAN-silent, repeatable.

Blocked: Cross-cache write (into spray object) requires forcing kfree(vvs) before workqueue runs — not achievable on single-CPU QEMU with the current trigger sequence.

Strongest path forward: Exfiltrate the UAF-read kernel pointers (sk_prot, sk_net) to userspace via a receiver-side socket operation — yielding KASLR bypass without needing the cross-cache step.
§ 06

AI-Augmented Research Pipeline

This research was conducted using an extended AI collaboration pipeline in which a large language model served as an active technical co-researcher throughout the entire process — from source reading and hypothesis formation, through GDB script generation, PoC development, and exploit primitive analysis. We document this methodology in detail because it represents a novel and reproducible approach to kernel vulnerability research.

Pipeline Architecture

The workflow was entirely human-directed: the researcher controlled all decisions about what to investigate, what to run, and what the evidence meant. The AI operated as a stateful technical assistant with continuous access to source code, GDB output, and PoC results shared iteratively across the session. The model was never given root access, never ran code, and never made independent decisions — it reasoned, proposed, and explained.

The Iterative Feedback Loop

Each research iteration followed a tight cycle: the researcher shared GDB output or PoC results as raw text; the AI read the output precisely (e.g., identifying that vvs=0xffff88800dbc5f00 in the watchpoint and vvs=0xffff88800dbc5f00 in get_credit were the same address, meaning the freed slot was reclaimed by a new socket rather than by the spray), then issued a precise diagnosis and corrected the next PoC version.

The most significant example: after the listener thread interference was identified as stealing the freed slab slot (a new vsock_create_connected() call reclaiming the freed vvs address), the AI proposed stopping the listener before freeing — which led to discovering that the volatile flag approach fails because accept() is a blocking syscall that doesn't check flags. This drove the exploration of the fast-path ordering problem, which ultimately defined the open research question.

What the AI Could Not Do

The model's limitations were equally instructive. It could not independently verify kernel addresses, could not run GDB, and had no direct access to the target system. When it proposed that the watchpoint was firing on the wrong vvs, that was a hypothesis — the researcher confirmed it by checking addresses. More subtly: the model initially suggested recv() before close() as a fast-path trigger, but careful source tracing revealed this cannot work — the workqueue delivers both the data SKBs and the peer's shutdown packet in the same run, defeating the ordering. The researcher's empirical GDB observations were the ground truth throughout.

Methodological Takeaways

Using an AI pipeline for kernel research introduces a fundamentally different workflow compared to manual analysis. Instead of a single researcher holding an entire call graph in their head, the model maintains working context across multiple files simultaneously — reading af_vsock.c, virtio_transport_common.c, and vsock_loopback.c in parallel and cross-referencing them on demand. This dramatically accelerates the hypothesis generation phase.

However, the pipeline succeeds precisely because the human researcher maintains epistemic control: every hypothesis is tested empirically, every GDB output is shared in full, and the researcher exercises judgment about which direction to pursue. The model serves as a force multiplier for the researcher's expertise — it does not replace the expertise itself. Kernel exploitation requires deep intuition about race conditions, slab allocator behavior, and workqueue scheduling that the model approximates from training but cannot truly verify without live system access.

§ 07

What We Learned About the Kernel

Socket Lifecycle is Non-Trivial

The gap between close(fd) and kfree(vvs) can be up to 8 seconds in the slow path. VSOCK_CLOSE_TIMEOUT = 8 * HZ is a protocol-level timeout, not a cleanup shortcut. During this window, the socket structure is unreferenced but not freed — a class of bugs distinct from classic UAF where the object is freed at syscall return time.

SKB Back-Pointers Are Not Reference-Counted

skb->sk is set in virtio_transport_alloc_skb() via skb_set_owner_w(), which takes a refcount on the socket. However, in the loopback path, consume_skb_sent(skb, false) does not consume the SKB (the receiver does) — and the refcount is released by sock_wfree (the SKB destructor) only when the SKB is finally freed by the receiver. The window between socket teardown and SKB freeing is the entire vulnerability surface.

SLUB Cache Behavior

The kmalloc-96 cache is shared across many subsystems. Freed objects are immediately returned to the per-CPU slab and available for reallocation by any kmalloc(96, GFP_KERNEL) call. With CONFIG_INIT_ON_ALLOC_DEFAULT_ON=n, freshly allocated objects carry stale data — eliminating one of the standard Popov-style msg_msg spray assumptions (that m_list fields are zero immediately post-allocation).

Workqueue Scheduling on Single-CPU QEMU

Bounded workqueues (flags=0) run on worker threads that are scheduled by the CFS scheduler. On a single-CPU system, a workqueue worker and a userspace thread cannot run simultaneously. After close(fd), the kernel processes the syscall return, re-enters userspace, and only then does the scheduler select the workqueue worker. sched_yield() explicitly hands control to the worker. This determinism — while frustrating for ordering attacks — actually makes the UAF more reliably reproducible than on SMP systems.

cancel_pkt as a Missing Safety Net

The loopback transport's cancel_pkt implementation correctly calls virtio_transport_purge_skbs() to drain in-flight SKBs. The bug is that this is never called during normal socket close — only during connection timeout and explicit shutdown. A minimal fix would be to call transport->cancel_pkt(vsk) inside __vsock_release() before invoking transport->release(vsk), ensuring no SKBs remain in the loopback queue with a back-pointer to the socket being torn down.

§ 08

Conclusion

We identified, confirmed, and deeply analyzed a use-after-free vulnerability in the Linux kernel's vsock loopback transport (net/vmw_vsock/virtio_transport_common.c, virtio_transport_consume_skb_sent()), present in kernel 6.12.x.

The bug yields a KASAN-silent, controlled, repeatable write primitive on a kernel object at a known offset, alongside a UAF read of kernel text and data pointers from a refcount=0 socket structure. The fundamental exploitation challenge — reversing the ordering so the cross-cache write fires into a spray object rather than a live vvs — is an open problem constrained by the single-CPU workqueue scheduling behavior and the absence of a userspace-accessible mechanism to defer or cancel the loopback workqueue.

The research demonstrates that AI-augmented kernel analysis is a practically viable and powerful methodology, capable of identifying, tracing, and characterizing complex multi-file kernel race conditions at a depth and speed that significantly exceeds purely manual analysis — provided the human researcher maintains empirical control and treats every model-generated hypothesis as a testable claim rather than a fact.

Responsible Disclosure
This research was conducted as part of kernelCTF, a competition specifically designed for kernel vulnerability research. Findings were handled in accordance with the kernelCTF disclosure process. No production systems were targeted. All experiments were performed in isolated QEMU virtual machine environments.
